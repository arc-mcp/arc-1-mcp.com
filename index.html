Why ARC-1

AI for SAP needs an architecture, not another laptop script.

SAP development is not like editing a local web app. The useful context lives in ABAP objects, packages, transports, dumps, ATC findings, CDS metadata, authorizations, tables, and system release state. ARC-1 is built for that reality: controlled context first, governed action second.

Central control

Run one managed MCP endpoint per SAP system instead of letting every client, IDE, and user invent their own credential path.

Safe defaults

Read-only by default. Writes, table preview, SQL, transports, Git operations, and packages are explicit opt-ins.

Real identity

Use XSUAA, OIDC, API keys, BTP destinations, and Principal Propagation so SAP still sees the right user where your landscape supports it.

Auditable by design

Structured audit events can go to stderr, file, or the BTP Audit Log Service. The system can answer who called what.

Product position

The CIO case for ARC-1

ARC-1 is not positioned as a prompt trick. It is an open, inspectable access layer for SAP AI adoption. It lets teams start with read-only value, prove the control model, then open carefully selected write paths only where the operating model can support them.

Lower adoption risk Use existing SAP authorization, BTP identity, role collections, and audit services.
Keep client choice open Serve Claude, Copilot, Joule, Cursor, VS Code, Eclipse, and future MCP clients from one endpoint.
Protect productive systems Keep production read-only, allow sandbox and DEV writes selectively, and review every capability.
Avoid black boxes The core product is open source, testable, and designed around explicit security gates.

What ARC-1 is good for

Use cases that matter beyond a developer demo

The first value is read access: current system state, source, metadata, checks, dependencies, dumps, and transport context. Write access can come later, in restricted systems and packages, after the organization is comfortable with the control model.

ABAP development assistance

Read source, search objects, edit methods, activate, run syntax checks, run ABAP Unit, inspect ATC, and keep the AI grounded in the actual SAP system.

Clean Core and modernization

Analyze custom code against release state and SAP guidance, identify direct table usage, build modernization backlogs, and guide SEGW-to-RAP moves.

Architecture and impact analysis

Turn change requests into technical risk views: dependencies, activation order, table impact, transport requirements, and affected objects.

Support and diagnostics

Connect weak tickets to ST22 dumps, source code, gateway messages, traces, package contents, and likely root causes before a developer opens the IDE.

Microsoft 365 and Joule agents

Reuse the same BTP-deployed ARC-1 endpoint from Copilot Studio, Teams, Microsoft 365 Copilot, and Joule Studio without creating a separate SAP access model.

GitHub workflow automation

Give PR checks and AI reviewers live SAP context for ABAP Unit, ATC, activated-source drift checks, and read-only code review.

Architecture

One controlled SAP access layer for many AI clients

ARC-1 can run locally, in Docker, or as a BTP Cloud Foundry app. The enterprise pattern is central: one endpoint, one safety ceiling, one audit path, and SAP authorizations still deciding what the backend user may do.

Gate 1

Server ceiling

Admin-controlled flags define what this ARC-1 instance can ever do.

Gate 2

User permission

Scopes, role collections, OIDC claims, or API-key profiles restrict the user.

Gate 3

SAP authorization

SAP still enforces S_DEVELOP, package, transport, table, and system permissions.

Effective permission = server ceiling AND user permission AND SAP authorization

Start small

Install locally in minutes. Deploy centrally when the team is ready.

ARC-1 ships as an npm package, Docker image, Claude Desktop extension, Claude Code plugin, and BTP-ready application. The right first pilot is usually read/search/diagnose against DEV or a sandbox.
